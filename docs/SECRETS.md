# Secrets Management

This document describes how secrets are handled in the eBillio FreeRADIUS
project: where they live, how to generate them, how to rotate them, and what
to do when (not if) one leaks.

---

## 1. Security model

| File            | Tracked in git? | Contains real secrets? |
| --------------- | --------------- | ---------------------- |
| `.env`          | No (gitignored) | Yes                    |
| `.env.example`  | Yes             | No - placeholders only |
| `docker-compose.yml` | Yes        | No - references `${...}` only |
| `freeradius/**` | Yes             | No - values come from env |
| `sql/**`        | Yes             | No                     |

Rules:

1. `.env` MUST NEVER be committed. It is listed in `.gitignore`.
2. `.env.example` MUST only contain `CHANGE_ME_*` placeholders or obviously
   fake values. Never paste a real secret here "just for testing".
3. Every secret MUST be generated by a cryptographically secure RNG. No
   human-chosen passwords, no dictionary words, no dates, no project name
   mashups (`Eb1ll10_R4d1us_2026!`-style values are forbidden).
4. Every environment (dev, staging, prod) gets DIFFERENT secrets. Never copy
   prod secrets into dev.
5. Secrets are only ever shared through an approved secret manager (1Password,
   Bitwarden, HashiCorp Vault, AWS Secrets Manager, etc.) - never via Slack,
   email, SMS, ticket comments, or screenshots.

## 2. The variables

All secrets are injected via `docker-compose.yml` from `.env`.

| Variable           | Consumer        | Purpose                                       | Min length |
| ------------------ | --------------- | --------------------------------------------- | ---------- |
| `RADIUS_SECRET`    | FreeRADIUS      | Shared secret between RADIUS server and NAS   | 32 (40+ recommended) |
| `DB_ROOT_PASSWORD` | MySQL           | `root@%` admin password                       | 32         |
| `DB_USERNAME`      | MySQL, FreeRADIUS | Application DB user (not a secret)          | n/a        |
| `DB_PASSWORD`      | MySQL, FreeRADIUS | Application DB user password                | 32         |

### Generating a fresh value

```bash
openssl rand -base64 32 | tr -d '/+=' | head -c 40
```

This yields 40 URL-safe alphanumeric characters with ~238 bits of entropy -
safe for both RADIUS (which has historical quirks with some punctuation) and
MySQL.

If you need a value with symbols for non-RADIUS use:

```bash
openssl rand -base64 48 | head -c 48
```

## 3. The 2026-04-10 leak - what happened and what to do

Commit `53679ea` ("Initial commit: FreeRADIUS 3.2.7 + MySQL 8.0 Docker stack")
committed a `.env` file containing real-looking production credentials:

```
RADIUS_SECRET=Eb1ll10_R4d1us_2026!
DB_ROOT_PASSWORD=R00t_Rad1us_2026
DB_PASSWORD=Rad1us_Pr0d_2026
```

### These values are now PERMANENTLY COMPROMISED

Anyone who cloned, forked, mirrored, or received a bundle of this repo before
the fix has these values. They will remain in git history even after we
rewrite it, because:

- Any existing clone still has them.
- Any backup, CI cache, or archived PR still has them.
- GitHub/GitLab keep dangling commits for a window (sometimes indefinitely
  via fork networks or commit SHA URLs).
- GitHub search and code-scanning bots may have already indexed them.

### Mandatory rotation checklist

Do ALL of the following, in order, before treating the incident as closed:

1. [ ] Rotate `RADIUS_SECRET` in every deployed FreeRADIUS instance.
2. [ ] Rotate the matching `secret = ...` entries on EVERY NAS / BRAS /
       access point that talks to this RADIUS server (`clients.conf`
       counterparts). Coordinate a short window with the network team - any
       NAS still using the old secret will fail authentication until updated.
3. [ ] Rotate `DB_ROOT_PASSWORD` on every MySQL instance:
       ```sql
       ALTER USER 'root'@'%' IDENTIFIED BY '<new value>';
       ALTER USER 'root'@'localhost' IDENTIFIED BY '<new value>';
       FLUSH PRIVILEGES;
       ```
4. [ ] Rotate `DB_PASSWORD` on every MySQL instance:
       ```sql
       ALTER USER 'radius'@'%' IDENTIFIED BY '<new value>';
       FLUSH PRIVILEGES;
       ```
       Then update `.env` on the FreeRADIUS host and restart the container.
5. [ ] Audit MySQL `general_log` / FreeRADIUS `radius.log` for any
       unexpected source IPs around the exposure window.
6. [ ] Audit `radacct` for suspicious sessions (unusual NAS-IP, unusual
       Framed-IP, abnormally long sessions, Calling-Station-Ids you don't
       recognize).
7. [ ] If the MySQL port (3306) was ever exposed to the internet, assume
       data exfiltration and run a full `radcheck` / `radreply` audit.
8. [ ] Store the new values in the team secret manager, not in chat.
9. [ ] Notify anyone who may have pulled the bad commit that they should
       delete their local clone and re-clone after history rewrite (if you
       choose to rewrite).

## 4. Scrubbing git history (optional, destructive)

History rewriting is NOT a substitute for rotation. Rotate first. Only then
consider rewriting history to reduce the blast radius of future clones.

WARNING: Rewriting history changes every commit SHA from the rewrite point
forward. Every collaborator MUST delete their local clone and re-clone.
Open PRs will need to be rebased or recreated. CI caches keyed on SHA will
miss. Do not do this on a Friday.

### Option A: `git filter-repo` (recommended)

```bash
# Install (Debian/Ubuntu):
sudo apt install git-filter-repo
# or:  pip install git-filter-repo

# From a FRESH clone (filter-repo refuses to run on a non-fresh clone):
git clone --mirror git@github.com:ebillio/radius.git radius-scrub.git
cd radius-scrub.git

# Remove the file from all history:
git filter-repo --path .env --invert-paths

# Also redact any stray string occurrences (belt and braces):
cat > ../replacements.txt <<'EOF'
Eb1ll10_R4d1us_2026!==>REDACTED
R00t_Rad1us_2026==>REDACTED
Rad1us_Pr0d_2026==>REDACTED
EOF
git filter-repo --replace-text ../replacements.txt

# Force push the rewritten history:
git push --force --all
git push --force --tags
```

### Option B: BFG Repo-Cleaner

```bash
# Download BFG from https://rtyley.github.io/bfg-repo-cleaner/
# Requires Java.

git clone --mirror git@github.com:ebillio/radius.git radius-scrub.git

java -jar bfg.jar --delete-files .env radius-scrub.git

cat > replacements.txt <<'EOF'
Eb1ll10_R4d1us_2026!
R00t_Rad1us_2026
Rad1us_Pr0d_2026
EOF
java -jar bfg.jar --replace-text replacements.txt radius-scrub.git

cd radius-scrub.git
git reflog expire --expire=now --all
git gc --prune=now --aggressive
git push --force --all
git push --force --tags
```

### After either option

1. Ask GitHub/GitLab support to purge cached views of the offending commits
   and to invalidate the old SHAs from search. GitHub has a specific form
   for this.
2. Tell every collaborator to delete their local clone and re-clone.
3. Invalidate all forks (or ask fork owners to do the same).
4. Re-run the rotation checklist in section 3 just to be sure.

**These commands are documented. DO NOT run them without a coordinated plan
with the whole team.**

## 5. Ongoing rotation cadence

| Secret           | Routine rotation | Emergency rotation triggers                        |
| ---------------- | ---------------- | -------------------------------------------------- |
| `RADIUS_SECRET`  | Every 12 months  | Any NAS compromise, staff offboarding with access, suspected leak |
| `DB_ROOT_PASSWORD` | Every 6 months | Suspected leak, staff offboarding, before major audits |
| `DB_PASSWORD`    | Every 6 months   | Suspected leak, staff offboarding                  |
| TLS certificates | Per cert lifetime (<= 1 year) | Key compromise                        |

### RADIUS_SECRET rotation - coordinated procedure

Because the RADIUS shared secret is symmetric and lives on both the server
and every NAS, a rotation is a change window, not a one-sided action.

1. Generate the new secret (`openssl rand -base64 32 | tr -d '/+=' | head -c 40`).
2. Store it in the team secret manager.
3. Notify the network team and schedule a maintenance window.
4. During the window:
   a. Update `.env` on the RADIUS host and `docker compose up -d` so
      FreeRADIUS picks up the new value.
   b. Update the `secret = ...` on every NAS in parallel (Ansible, RANCID,
      vendor controller, etc.).
   c. Watch `radius.log` for `Dropping packet without response because of
      error: Received packet from X with invalid Message-Authenticator` -
      those are NASes still on the old secret.
5. Once all NASes are green, purge the old secret from the secret manager.
6. Record the rotation in the change log.

### DB_PASSWORD rotation - zero-downtime procedure

1. Generate new value.
2. `ALTER USER 'radius'@'%' IDENTIFIED BY '<new>';` on MySQL.
3. Update `.env` on the FreeRADIUS host.
4. `docker compose up -d freeradius` to restart with the new value.
5. Confirm auth traffic is still flowing.

## 6. CoA (Change-of-Authorization) secrets

CoA/Disconnect-Request (RFC 5176, UDP/3799) uses its own shared secret
between the CoA *client* (your RADIUS server or a billing app) and the NAS
acting as the CoA *listener*.

**CoA shared secrets MUST be different from the auth/acct `RADIUS_SECRET`.**

Reasons:

1. **Blast radius.** If the auth/acct secret leaks, an attacker who can
   reach UDP/3799 on your NAS should NOT automatically be able to forge
   disconnect or bandwidth-change packets for live subscribers.
2. **Rotation independence.** CoA is typically issued from a different
   process (a billing/provisioning app) with a different deployment
   cadence. Separate secrets let you rotate one without a flag day on the
   other.
3. **Auditing.** Separate secrets mean separate log lines and easier
   attribution when something goes wrong.
4. **Least privilege.** Not every system that needs to send Access-Request
   needs to be able to forcibly disconnect paying customers.

Naming convention for future `.env` additions (another agent is enabling
CoA):

```
RADIUS_SECRET=...         # auth + acct shared secret (this file, today)
COA_SECRET=...            # CoA / Disconnect shared secret (new)
```

Add `COA_SECRET` to `.env.example` as a `CHANGE_ME_*` placeholder when CoA
is wired up, and add it to the rotation table in section 5 with the same
cadence as `RADIUS_SECRET`.

## 7. Pre-commit hygiene (recommended)

To stop the next leak before it happens, install a pre-commit secret scanner:

```bash
# gitleaks
brew install gitleaks    # or: https://github.com/gitleaks/gitleaks
gitleaks protect --staged

# or pre-commit framework with detect-secrets
pip install pre-commit detect-secrets
# add a .pre-commit-config.yaml with the detect-secrets hook
pre-commit install
```

CI should also run `gitleaks detect` on every PR and fail the build on any
finding.
